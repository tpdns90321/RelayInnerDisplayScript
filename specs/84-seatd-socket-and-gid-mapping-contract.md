# Spec 84. Seatd Socket and GID Mapping Contract

## Context / Problem

The LXC direct-seat design keeps the compositor inside an unprivileged container while host `seatd` mediates access to graphics, input, and the virtual terminal. The default `seatd` socket is owned by a host user/group and is not automatically accessible to an unprivileged LXC user because container IDs map to unprivileged host IDs.

This spec defines a narrow group-mapping contract so the container's `relayinner-display` runtime group can access the host `seatd` socket without making the socket world-writable or converting the CT to privileged mode.

## Goals / Non-goals

Goals:

- Use the host default `/run/seatd.sock` as the LXC seatd socket path.
- Run host `seatd` with a relay-specific socket group.
- Map exactly one host group into the unprivileged LXC for socket access.
- Keep the container runtime user/group name as `relayinner-display`.
- Track all group and idmap changes for uninstall.

Non-goals:

- Building a custom `seatd` with a relay-specific socket path.
- Making `/run/seatd.sock` world-writable.
- Mapping broad host group ranges into the container.
- Supporting multiple LXC guests sharing the same seatd socket in the MVP.
- Running a second `seatd` inside the container.

## User stories

- As an operator, I want the LXC compositor to connect to host `seatd` without weakening socket permissions for the whole host.
- As a maintainer, I want a deterministic mapping between one host group and one container group.
- As a reviewer, I want idmap changes to be explicit, reversible, and limited.

## Public API / Interfaces

Host group naming convention:

```text
relayinner-display-lxc
```

Container runtime group:

```text
relayinner-display
```

Host `seatd` command shape:

```sh
seatd -g relayinner-display-lxc
```

The host profile renders an idmap exception that maps the host GID for `relayinner-display-lxc` to the container GID for `relayinner-display`. The exact `lxc.idmap` lines depend on existing subordinate ID ranges and must be generated by the host profile rather than hand-written in docs.

The container kiosk environment uses:

```ini
Environment=LIBSEAT_BACKEND=seatd
```

The default socket path is used unless later implementation proves a different path is required inside the container.

## Data model / Persistence

Install-state must store:

- host group name
- host group GID
- container group name
- container group GID expected by the profile
- generated idmap lines
- previous or created host group state
- previous `seatd` service command or relay-rendered service content
- LXC config backup path and managed block

The state must distinguish between a group created by the installer and a pre-existing group reused by the installer.

## Security model (Permission/Isolation/Audit)

Permissions:

- Socket access is granted through group permission only.
- The host group is dedicated to the relay LXC seat path.
- The container is granted no additional host groups beyond the single mapping required for `seatd`.

Isolation:

- The design assumes one relay LXC owns the physical appliance seat.
- Existing host desktop or unrelated seatd consumers may be incompatible with this appliance profile.
- The host-direct profile must not be silently converted to this group model unless explicitly selected.

Audit:

- Dry-run must print host GID, container GID, and generated idmap lines.
- Apply mode must print whether it created or reused the host group.
- Uninstall must print whether the group was removed, preserved, or left for manual cleanup.

## Acceptance criteria (Testable, Verifiable)

- Host bootstrap creates or reuses `relayinner-display-lxc` with a stable GID.
- Host `seatd` is rendered or configured to own `/run/seatd.sock` with the relay LXC group.
- LXC config contains a single GID mapping exception for the seatd group path.
- The container `relayinner-display` group can connect to `/run/seatd.sock`.
- The socket is not world-writable.
- The CT remains unprivileged.
- Dry-run shows all idmap changes before apply.
- Uninstall restores the previous seatd service behavior and removes idmap changes from the managed block.

## Test plan

Automated tests should cover:

- host group creation versus reuse
- idmap rendering with fake subordinate ranges
- conflict detection for existing idmap lines
- install-state serialization for group ownership
- uninstall behavior for created and reused groups
- refusal to render world-writable socket policy

Manual validation:

1. Apply the host profile to a stopped CT.
2. Start host `seatd` through the rendered service.
3. Confirm `/run/seatd.sock` group ownership.
4. Start the CT.
5. Confirm the container runtime user can connect through libseat.
6. Confirm unrelated container users cannot connect unless in the mapped group.

## Rollout / Backward compatibility

Host-direct installs continue using the existing host runtime group unless the LXC host profile is selected. The LXC profile may change host `seatd` ownership, so docs must describe that this is an appliance seat ownership mode.

Existing install-state without group mapping records must uninstall through the previous host-direct path without errors.

## Open questions

- What exact idmap algorithm best handles hosts with existing non-default subordinate GID ranges?
- Should the installer reserve a fixed host GID range or always allocate dynamically?
- How should the profile detect and report an existing system `seatd` service not owned by this project?
